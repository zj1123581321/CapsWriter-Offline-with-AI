# 下游客户端接入指南（WebSocket 协议 + 多模型适配）

面向要接入本服务端做转录/字幕的下游开发者。涵盖 WS 协议、音频格式、**字级时间戳的正确用法**，以及一个真实踩坑（不同底层模型的 token 形态差异导致时间轴整体压缩 ~10%）的复盘与规避。读完可在任意语言里写出稳定接入、且**自动适配 paraformer / qwen3 等不同底层模型**的客户端。

> 协议真相源：`core/protocol.py`（`AudioMessage` / `RecognitionMessage`）、`core/server/connection/ws_recv.py`、`core/server/worker/pipeline.py`。本文基于 `deploy/linux-vulkan` 分支（Qwen3-ASR + force-aligner）实测整理。

## 0. 服务端部署（GPU 可选）

主线默认全 CPU，开箱即用。若自建服务端要上 GPU（Linux + CUDA/Vulkan），**无需改源码**，启动前设环境变量即可：

```bash
export CW_ONNX_PROVIDER=CUDA   # ONNX 后端：CPU(默认)/CUDA/DML
export CW_LLM_USE_GPU=1        # GGUF 解码器用 GPU
python start_server.py
```

完整变量表与降级说明见 [显卡加速的若干问题.md](显卡加速的若干问题.md#二onnx-组件加速)。

## 1. 连接

- URL：`ws://<host>:<port>`（默认端口见 `config_server.py`，如 `6016`）。**无 subprotocol、无握手、无前置 config**——连上直接发音频帧。
- 建议连接参数：`max_size=None`（音频帧大，别限制帧大小）、`ping_interval=None`（长音频一次性上行时服务端忙于收/处理，默认 ping 超时会误杀连接，活性改由下游自己的 stall 超时兜底）。

## 2. 发送：`AudioMessage`（一条 WS **文本帧** = 一个 JSON）

字段（`core/protocol.py`，`from_dict` 对必填项用 `data['x']` 强取，缺一即崩 → 服务端 handler 静默异常、零回包）：

| 字段 | 必填 | 类型 / 默认 | 说明 |
|---|---|---|---|
| `task_id` | ✅ | str | 一次转录任务唯一 id（uuid 字符串） |
| `source` | ✅ | `"file"` / `"mic"` | 文件转录用 `"file"` |
| `data` | ✅ | str | base64(裸 PCM)，见下 |
| `is_final` | ✅ | bool | 最后一帧置 `true` 通知收尾 |
| `time_start` | ✅ | float | 任务起始时间戳（`time.time()`） |
| `seg_duration` | ➖ | float = 15.0 | 服务端分段识别窗口（秒） |
| `seg_overlap` | ➖ | float = 2.0 | 分段滑动窗口重叠（秒） |
| `context` | ➖ | str = `""` | 提示上下文 |
| `language` | ➖ | str = `"auto"` | 语言 |

> ⚠️ 不要发协议外的字段（例如 `time_frame`）。`AudioMessage` 没有它；多发无用、徒增歧义。

**音频格式（`data` 解码后）**：`float32 小端 / 16000Hz / 单声道 / 裸 PCM`，**无 WAV 头**。每秒 `16000 × 4 = 64000` 字节。用 ffmpeg 抽：

```
ffmpeg -nostdin -i <file> -f f32le -ac 1 -ar 16000 -
```

**长音频用流式分块**：把 PCM 切成固定大小帧（官方客户端 `core/client/transcribe/file_transcriber.py` 用 **1 分钟/帧** = `16000*4*60` 字节）连续发，无需等回复，最后一帧 `is_final=true`。服务端在 `ws_recv.py` 内部按 `seg_duration` 缓冲分段、滑动窗口识别、自动拼接多段结果。

## 3. 接收：`RecognitionMessage`（JSON）

| 字段 | 说明 |
|---|---|
| `task_id` | 对应请求 |
| `is_final` | `true` = 全部识别完成，**收到即结束**（之前是中间结果，可忽略或用于实时展示） |
| `text` | 简单拼接文本 |
| `text_accu` | 基于时间戳精确去重拼接（推荐用它） |
| `tokens` | **字级 token 数组** |
| `timestamps` | **与 `tokens` 平行的字级起始时间（秒）** |
| `duration` / `time_*` | 已处理时长、各阶段时间戳 |

**时间戳来源**：Qwen3-ASR 出文本后，force-aligner（`core/server/engines/force_aligner_gguf/`）把文本对齐回音频，给每个 token 一个起始时间。字级、单位秒、~80ms 精度，**已累加全局偏移**（流式多段由服务端 `pipeline.py` 拼接），即拿到的 `timestamps` 就是相对整个文件的绝对时间。

## 4. ⚠️ 核心踩坑：token 形态因底层模型而异（做字幕跟随必读）

要做「点句跳秒 / 当前句高亮跟随」，需要**句级时间戳**。这里有个真实事故：**不同底层模型的 `tokens` 形态不同，下游若用错方式从文本反推时间戳，会让整条时间轴线性压缩 ~10%**（23 分钟视频末句偏早 140 秒）。

### 两种典型 token 形态

| 模型 | `tokens` 形态 | 标点 | 数字 |
|---|---|---|---|
| **paraformer-zh** | 纯单字 | **不在 tokens**，只在 `text` | 逐字 |
| **qwen3-asr-1.7b** | 字级 | **标点也是独立 token**（如 `，` `。` `？`） | 多字符 token（如 `"1971"` `"9250"`） |

### 错误做法（会压缩时间轴）

❌ 「拿 `text` 自己切句 → 把句子**去标点**后的字符数当偏移 → 映射回 `tokens` 取时间戳」。

- 在 paraformer 上**碰巧正确**（token 无标点，去标点字符数与 token 流一一对应）。
- 在 qwen3 上**逐标点错位**：text 去掉的每个标点，在 tokens 里却占一个 token 位置。错位随标点数累积——实测 9478 token 含 871 个标点 token，末尾错位 `871/9478 ≈ 9.2%`，正好压缩 ~10%。

### 正确做法

✅ **永远直接使用服务端给的 `(token, timestamp)` 对，不要从 text 反推映射回 token。** 句子的起止时间 = 该句首/末 token 自带的 `timestamp`。

✅ **自动适配 token 形态**（下游零配置，上游随便换模型）：检测 `tokens` 里有没有标点 token——

- **有标点 token（qwen3）**：直接遍历 `tokens`，在标点 token 处断句，每句首/末 token 的 `timestamp` 即 start/end。
- **无标点 token（paraformer）**：用 `text` 的标点切句，按**去标点字符数**映射回单字 token 取 `timestamp`（此形态下映射不会错位）。

```python
PUNCT = set("。！？!?，,、；;")

def has_punct_tokens(tokens):
    return any(t in PUNCT for t in tokens)

# qwen3 路径：按标点 token 断句，时间戳直接取
def segments_by_token(tokens, timestamps):
    segs, cur, start = [], [], None
    for tok, ts in zip(tokens, timestamps):
        if start is None: start = ts
        cur.append(tok)
        if tok in PUNCT:
            segs.append({"start": start, "end": ts, "text": "".join(cur)})
            cur, start = [], None
    if cur:
        segs.append({"start": start, "end": timestamps[-1], "text": "".join(cur)})
    return segs

# 入口：自动分发
def make_segments(text, tokens, timestamps):
    if has_punct_tokens(tokens):
        return segments_by_token(tokens, timestamps)      # qwen3
    else:
        return segments_by_text_mapping(text, tokens, timestamps)  # paraformer（text 标点切 + 单字映射）
```

> 一个完整、带「短句优化 / 碎片合并 / 无标点长串兜底拆分 / 自动分发」的参考实现，见下游项目 tg-archiver 的 `src/tg_archiver/asr/segments.py:create_segments`（MIT 风格，可直接借鉴）。

## 5. 最小可用客户端（Python，裸 WS，约 40 行）

```python
import asyncio, base64, json, time, uuid, subprocess
import websockets

URI = "ws://127.0.0.1:6016"
FRAME = 16000 * 4 * 60  # 1 分钟 f32le

def pcm_stream(path):
    p = subprocess.Popen(
        ["ffmpeg", "-nostdin", "-i", path, "-f", "f32le", "-ac", "1", "-ar", "16000", "-"],
        stdout=subprocess.PIPE)
    while True:
        b = p.stdout.read(FRAME)
        if not b:
            break
        yield b

async def transcribe(path):
    tid, t0 = str(uuid.uuid4()), time.time()
    async with websockets.connect(URI, max_size=None, ping_interval=None) as ws:
        async def send():
            frames = list(pcm_stream(path))
            for i, frame in enumerate(frames):
                await ws.send(json.dumps({
                    "task_id": tid, "source": "file",
                    "data": base64.b64encode(frame).decode(),
                    "is_final": (i == len(frames) - 1),
                    "time_start": t0, "seg_duration": 15.0, "seg_overlap": 2.0,
                    "context": "", "language": "auto",
                }))
        asyncio.create_task(send())
        while True:
            msg = json.loads(await ws.recv())
            if msg.get("is_final"):
                return msg  # 含 text / text_accu / tokens / timestamps

m = asyncio.run(transcribe("input.mp4"))
print(m["text_accu"])
print("末 token 时间:", m["timestamps"][-1])  # 应 ≈ 音频真实时长
```

**自检**：流式发完，`timestamps[-1]` 应 ≈ 音频真实时长（`ffprobe -show_entries format=duration`）。若明显偏小（如短 10%），多半是下游用了「去标点反推映射」的错误做法，回看第 4 节。

## 6. 模型选型对照

| 模型 | 资源 | 质量 | 时间戳 |
|---|---|---|---|
| **paraformer-zh** | CPU 即可，快 | 一般 | 较粗 |
| **qwen3-asr-1.7b** | 需 GPU / MPS | 高 | force-align，准（~80ms） |

下游只要按第 4 节自动适配 token 形态，**切换底层模型时客户端代码无需改动**，只换连接地址即可。

## 7. 常见错误速查

- **零回包 / 一直无响应**：多半缺必填字段（`source` / `time_start` 最易漏）→ 服务端 `from_dict` KeyError、handler 静默崩。
- **时间轴整体压缩 ~10%（越后越偏）**：下游用「text 去标点反推映射回 token」，遇含标点 token 的模型（qwen3）。改为直接用 `(token, timestamp)`，见第 4 节。
- **音频不识别 / 结果乱**：检查音频是否为 `f32le / 16k / mono` 裸 PCM、无 WAV 头。
- **连接被中途断开**：长音频一次性上行时关掉客户端 ping（`ping_interval=None`），用业务侧 stall 超时兜底。
